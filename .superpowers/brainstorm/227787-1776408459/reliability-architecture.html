Section 1 — Reliability Shield Architecture

Goal: make StockPilot dependable when upstream market data is flaky, without adding heavy infra.

Proposed request path
Callers
CLI · Web · Backtest · Agent Graph · News
↓
Single entrypoint
Data Gateway
one place for price, fundamentals, news, screening
↓
Reliability shield
1. Cache layer (SQLite/files) — fresh hit first
2. Source health tracker — failure counts, cooldown, recovery
3. Fallback router — try alternates by market/instrument
4. Normalized error envelope — never leak raw adapter chaos upward
↓
Existing adapters remain
AKShare · yfinance · future adapters
wrapped, not rewritten
Behavior under failure
Fresh success
Return fresh data + cache metadata
Primary source down
Mark source unhealthy → try fallback source
No source succeeds but cache exists
Return stale data with explicit fresh/stale/unavailable status
Nothing available
Return clear domain error (for example 503 data unavailable), not scattered adapter exceptions
Why this shape

Pros

  • Improves reliability everywhere at once
  • Preserves current adapter investment
  • Simple local-first operations model

Trade-off

  • Needs strict metadata and cache rules
  • Does not solve all freshness issues on day one

Key boundary

Adapters keep doing source-specific fetching. The new layer owns reliability policy, caching policy, and error normalization.
