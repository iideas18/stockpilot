Section 3 — Source Health, Fallbacks & Error Contract

The system should not blindly retry dead sources or leak source-specific chaos upward.

Source health model

Healthy

Normal priority routing

Degraded

Still callable, but lower priority and monitored closely

Cooling down

Temporarily skipped after repeated failures or rate-limit responses

Recovered

Probe succeeded, source can re-enter normal rotation

Tracked per source + market + data type
Example: akshare / a_share / daily_price can be unhealthy while yfinance / us / daily_price remains healthy
Fallback routing policy
Router input
market + instrument + request type + freshness need
Selection rule
pick highest-ranked healthy source first, then step down through ranked alternates
Failure response
increment failure count, set cooldown, record source-specific reason
Recovery
allow periodic probe or timed re-entry instead of permanent failure state
Important limit
no endless cascade; define max attempts per request to keep latency predictable

Normalized error envelope

What callers should receive
status: fresh | stale | unavailable
reason: timeout | rate_limited | parse_failed | upstream_unavailable
source_attempts: ordered list of tried sources
retry_hint: optional cooldown / suggested retry window
user_message: short safe message for CLI/web
debug_message: structured technical detail for logs

Why this helps

  • Backtests stop failing randomly because one source blips
  • Web/CLI can warn about stale data honestly
  • Agents can reason about data confidence instead of assuming perfect inputs

What it costs

  • Need a source registry and health metadata store
  • Need explicit ranking per market and data domain
  • Need consistent logging so failures are diagnosable
