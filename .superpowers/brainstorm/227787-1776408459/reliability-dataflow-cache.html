Section 2 — Request Flow & Cache Policy

Recommended behavior: prefer fresh data, tolerate stale data when upstream is down, and always tell the caller what quality level it received.

Request lifecycle
Step 1
Check in-memory cache for hot repeated reads in the same process
Step 2
Check persistent cache (SQLite/file store) by market + symbol + request shape
Step 3
If cache is fresh → return immediately
Step 4
If cache is stale → try upstream refresh through healthy sources
Step 5
If refresh succeeds → overwrite cache and return fresh
Step 6
If refresh fails but stale exists → return stale with explicit metadata
Step 7
If no data exists anywhere → return normalized unavailable error
Recommended cache classes

Realtime-ish quote

TTL: 30–120s
Use stale only briefly

Daily OHLCV

TTL: until next trading day close
Best fit for current StockPilot usage

Fundamentals / profile

TTL: 1–7 days
Low refresh urgency

News / sentiment

TTL: 15–60 min
Fresh enough without hammering sources

Cache record metadata
key · source · fetched_at · expires_at · freshness_state · market · request params · schema_version

Returned freshness contract

F

fresh

Upstream fetch or non-expired cache

S

stale

Expired cache used because refresh failed or was skipped

U

unavailable

No acceptable data found anywhere

Callers like CLI/web/agents should surface this state instead of pretending all data is equally trustworthy.

A

Looks right

Continue to source health, fallback policy, and error handling.

B

Needs revision

The cache rules or stale-data behavior should be changed before moving on.
